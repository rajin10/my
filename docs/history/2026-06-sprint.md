## New API modules

Added June 2026 as part of the sites ↔ backend gap-fill.

### Favourites (`/v1/favourites`)
- `GET /` — list authenticated user's saved venues
- `GET /:venueId` — check if a venue is saved (`{ isFavourited: boolean }`)
- `POST /:venueId` — add venue to favourites (409 if already saved)
- `DELETE /:venueId` — remove venue from favourites (204)
- Schema: `packages/core/src/database/schema/favourites.schema.ts`
- Repo: `packages/core/src/database/repositories/favourites.repository.ts`

### Staff Availability (`/v1/team/:id/availability`)
- `GET /:id/availability` — get 7-day schedule for a team member
- `PUT /:id/availability` — upsert schedule (one entry per `dayOfWeek` 0–6)
- Each slot: `{ dayOfWeek, isClosed, startTime, endTime }`
- Schema: `packages/core/src/database/schema/staff-availability.schema.ts`
- Requires owner/manager auth + `requireVenueStaff`

### Demo Requests (`/v1/demo-requests`)
- `POST /` — public endpoint; body `{ name, email, businessName, message? }`
- Schema: `packages/core/src/database/schema/demo-requests.schema.ts`
- Wired to the "Book a demo" CTA on `sites/marketing-site/for-business`

### Photo management
- `DELETE /v1/venues/:id/photos/:photoId` — remove a venue photo
- `PATCH /v1/venues/:id/photos/order` — body `{ orders: [{id, order}] }` reorders photos
- `DELETE /v1/services/:id/photo` — clears service photo (sets `imageUrl: null`)

### Analytics extensions
- `GET /v1/analytics/reviews` — avg rating, total reviews, star distribution
- `GET /v1/analytics/coupons` — redemptions + total discount per coupon
- `GET /v1/analytics/staff` — completed bookings + revenue per staff member
- All accept `venueId` + `range` (7|30|90) like existing analytics endpoints

### Bookings CSV export
- `GET /v1/bookings/export?venueId=&status=` — returns `text/csv` attachment
- Requires owner/manager auth; all branches for the venue are included

### Bookings list — venue-scoped query (2026-06 fix)
- `GET /v1/bookings` now returns `{ data: Booking[] }` (was raw array)
- `GET /v1/bookings/branch` now accepts `venueId` as an alternative to `branchId`; accepts optional `status` and `limit`; returns `{ data: Booking[] }`
- Both changes align the backend with the `PaginatedResponse<Booking>` contract the api-client and BD dashboard expect

### Search — cover photo + sort + map coords
- `GET /v1/search` results now include `coverPhotoUrl: string | null` — the first photo for each venue by display order
- `GET /v1/search` results include `lat` / `lng` from the first branch with coordinates (June 2026 v4)
- `sortBy=recommended|rating|price` is now wired in the marketing site search UI

### Branch availability (`/v1/branches/:id/availability`)
- `GET /?date=YYYY-MM-DD&serviceId=` — bookable ISO slot strings after hours + conflict checks (June 2026 v4)

### `@repo/api-client` additions
New endpoints: `api.favourites.*`, `api.staffAvailability.*`, `api.demoRequests.*`
New analytics: `api.analytics.reviews/coupons/staff`
New photo ops: `api.venues.deletePhoto/reorderPhotos`, `api.services.deletePhoto`
New export: `api.bookings.exportCsv(params)` → `Promise<Blob>` (use `getBlob` under the hood)
`EnrichedSearchResult` now includes `coverPhotoUrl: string | null`

## Feature-map audit & fixes

A full audit of backend ↔ frontend coverage was performed, resulting in [`feature-map.md`](../feature-map.md).

### What the audit found

The initial gap list (generated by single-line grep) was mostly **false positives**: multi-line method chains like

```js
api.services
  .delete(id)
```

were missed by `api\.\w+\.\w+` pattern matching. Manual source review confirmed that both owner platforms (`owner-app` and `business-dashboard`) already implement full parity across all modules: venues, services, coupons, team, reviews, campaigns, analytics, bookings, customers, staff availability, and notifications.

The customer platforms (`mobile-app` and `marketing-site`) were also verified to be at full parity after one fix (see below).

### Fixes applied

#### `useUploadServicePhoto` field name (business-dashboard)
- **File:** `sites/business-dashboard/src/hooks/useOwnerData.ts`
- **Bug:** `FormData.append("photo", file)` — the API reads `body.file` via `parseBody()`, so service photo uploads silently returned 422 "No file uploaded".
- **Fix:** Changed to `fd.append("file", file)`, matching the rule documented in the root `AGENTS.md` and the pattern already used by `useUploadVenuePhoto`.

#### Branch opening hours in `VenueScreen` (mobile-app)
- **File:** `apps/mobile-app/src/components/screens/VenueScreen.tsx`
- **Gap:** The marketing-site venue page showed per-branch opening hours (`api.branches.getHours`); the mobile venue screen did not.
- **Fix:** Added `useQueries` fetching `api.branches.getHours` for every branch in the venue (5 min stale time). The selected branch's weekly schedule is rendered between the branch selector and the services list, using the same `BranchHours` shape as the web.

## Low-priority gap fixes

### Archive venue (`venues.delete`)

Both owner platforms now expose a soft-delete ("archive") action for the venue. The API already enforced ownership; the frontends were simply missing the UI.

- **owner-app:** `VenueScreen` (`apps/owner-app/src/components/screens/MoreScreen.tsx`) — "Archive venue" button with `Alert.alert` destructive confirmation. Uses new `useDeleteVenue` hook in `apps/owner-app/src/hooks/useOwnerData.ts`.
- **business-dashboard:** `SettingsScreen` (`sites/business-dashboard/src/components/screens/SettingsScreen.tsx`) — "Danger zone" card with an Archive button, rendered via optional `onDeleteVenue` prop. Wired in `sites/business-dashboard/src/app/(dashboard)/settings/page.tsx`. Uses new `useDeleteVenue` hook in `sites/business-dashboard/src/hooks/useOwnerData.ts`.
- Archiving hides the venue from customer search. It can be restored via the existing Restore button on the same screen.

### Delete account (`users.delete`)

**API security fix:** `DELETE /api/v1/users/:id` had no authentication middleware — any caller could delete any user. Fixed by splitting `workers/api/src/modules/users/index.ts` into a `publicApp` (unauthenticated reads/writes) and a `selfApp` (delete, guarded by `authenticate`). The handler now asserts `c.var.user.id === id`; mismatches return 403. Tests updated to cover 401/403/200/404 cases.

Frontend "Delete account" added to all four customer/owner platforms:
- **mobile-app:** `AccountScreen` — button below Sign out, `Alert.alert` confirmation, calls `api.users.delete(authedUser.id)` then clears the query cache and signs out.
- **marketing-site:** `account/page.tsx` — button alongside Sign out in the profile card, `window.confirm`, then logout + token clear + redirect to `/`.
- **owner-app:** `MoreScreen` — "Delete account" button below Sign out (`danger` variant), `Alert.alert` confirmation. Fetches user ID via `useQuery(api.auth.me)` (5 min stale time); on confirm calls `api.users.delete(userId)` → `qc.clear()` → `signOut()`.
- **business-dashboard:** `account/page.tsx` — "Danger zone" card below the Sign out link, `window.confirm`, then deletes account and calls the existing `handleLogout()` flow (token clear + redirect to `/login`).

### Fresh booking status on mobile (`api.bookings.get`)

`BookingDetailSheet` (`apps/mobile-app/src/components/BookingDetailSheet.tsx`) now fires a background `useQuery` for `api.bookings.get(booking.id)` with `staleTime: 0` when the sheet opens. The fresh API status overrides the list-derived status for the pill and action buttons (Cancel / Leave review), so the sheet reflects updates delivered via push notification without requiring a full list refresh.

## Public-facing UX enhancements

Eight improvements shipped to `marketing-site` and `mobile-app`. All require no schema or API changes.

### 1 — Booking flow uses branch working hours (`/book/[venueId]/page.tsx`)
`generateSlots` now fetches `api.branches.getHours(activeBranch.id)` and constrains slots to the branch's `openTime`–`closeTime` for the selected day. Branches marked `isClosed` on that day return zero slots with an explanatory message. Falls back to 9 am–7 pm when no hours are configured.

### 2 — `next/image` replaced with `<img>` in search (`/search/SearchClient.tsx`)
`next/image` requires `width`/`height`/`fill` props that are unavailable for dynamic R2 URLs. Replaced with plain `<img>` throughout — consistent with the pattern already used in `venues/[id]`. **Rule:** never use `next/image` for remote/dynamic URLs in this project.

### 3 — Homepage VenueGrid skeleton (`VenueGrid.tsx`)
Loading state upgraded from 3 plain rectangles to 6 card-shaped skeletons (photo area + title + metadata rows + footer) matching the actual card layout.

### 4 — "Similar venues" section (`VenueClient.tsx`)
Bottom of venue detail page shows up to 3 venues in the same city + category via `api.search.venues({ category, city, limit: 4 })`. Current venue filtered out of results.

### 5 — VenueScreen loading skeleton (`VenueScreen.tsx`, mobile)
`detailQuery.isLoading` state now renders a gallery-height grey block + title/metadata/service-row skeletons instead of a plain "Loading venue…" text.

### 6 — Pull-to-refresh on FavouritesScreen (`FavouritesScreen.tsx`, mobile)
`RefreshControl` added to the `ScrollView`. On pull: `invalidateQueries(["favourites"])` + `invalidateQueries(["venue-photos"])`.

### 7 — Service photos in venue detail (web + mobile)
`Service.photoUrl?: string | null` added to `apps/mobile-app/src/data.ts`. `adaptService` in `src/lib/adapters.ts` maps `s.photoUrl`. Both `VenueClient.tsx` and `VenueScreen.tsx` render a `56×56` rounded thumbnail to the left of each service row when a photo URL is present.

### 8 — OG / Twitter meta tags on venue pages
`sites/marketing-site/src/app/venues/[id]/page.tsx` split into:
- `page.tsx` — server component; exports `generateMetadata` (fetches venue + first photo via `fetch`, returns `openGraph` + `twitter` fields)
- `VenueClient.tsx` — client component with all interactive logic

**Do not add `"use client"` to `page.tsx`** — it must remain a server component.

## API test coverage

Three modules added in the June 2026 sprint (`demo-requests`, `favourites`, `staff-availability`) had no route tests. Added 21 new tests across three new test files, bringing total coverage to **28 test files / 265 tests**.

These modules bypass the service layer and instantiate repositories directly in the route handler, so they use the **auto-mock + reinit** pattern rather than `createTestApp` service injection:

```ts
vi.mock("@repo/core/src/database/repositories/favourites.repository");
import { FavouritesRepository } from "...";

beforeEach(() => {
  vi.resetAllMocks();
  vi.mocked(FavouritesRepository).mockImplementation(function () {
    return { findByUser: vi.fn().mockResolvedValue([]), ... };
  } as never);
});
```

See [testing.md](../guides/testing.md) → "Auto-mock + reinit pattern" for the full explanation. `createTestApp` was also updated to register all three apps so their routes are reachable from any test.
