1 · System overview Stakeholder

Talash is a two-sided booking and discovery platform for the Bangladeshi wellness and beauty market. Customers use a mobile app and a marketing website to search for salons and spas (called venues), browse their services, book appointments, and leave reviews. Venue owners and their staff use a separate mobile app and a web dashboard to publish their venues, manage branches, services and working hours, confirm bookings, run promotions, and see analytics.

Behind these four apps sits a single backend running entirely on Cloudflare's global edge network: an HTTP API, a background job processor, and a scheduled-tasks worker, all sharing one database and a common code library. The platform exists to connect customers with local wellness providers and to give those providers the lightweight tooling they need to take and fulfil bookings — without anyone running their own servers.

4 · Key flows Engineer

The end-to-end processes that exercise the most of the system. Each shows the synchronous request and any asynchronous follow-up.

4.1 — Customer books an appointment

Booking lifecycle is a state machine: Pending → Confirmed → Completed, with Cancelled reachable from Pending/Confirmed. Rewards are credited only on complete; coupon usage is reverted on cancel.

4.2 — Google sign-in (two flows, one outcome)

Sessions are JWT-based: a short-lived access token (15 min) plus a rotating refresh token (30 days). The redirect flow uses a KV-stored state nonce for CSRF protection. These TTLs are read from the API worker guide; if config has since changed, re-confirm.

6 · Deployment view Stakeholder Engineer

How code gets from a developer's machine to users. The backend deploys automatically through GitHub Actions; the mobile apps are released manually.

Environments confirmed from CI config: a staging branch deploys to talash-api-staging and main deploys to talash-api-production; local dev uses Wrangler emulation. Whether additional preview/per-PR environments exist is unknown — see §8.
CI/CD specifics
  • PR → main: runs the checks job only (type-check, test, build). Lint is run locally; it is currently commented out in CI.
  • Push → main: checks then deploy, via cloudflare/wrangler-action@v3 per package.
  • Migrations: d1 migrations apply TALASH_DB --env production --remote runs before the API deploy, only when @repo/api is in scope. Idempotent; destructive migrations are split across two PRs.
  • Secrets: CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID. App secrets (JWT_SECRET, Google client id/secret) are Worker secrets, not in source.

7 · Glossary Stakeholder

Venue / Branch / Service
The domain hierarchy: a venue is a business (e.g. a salon brand); a branch is one physical location of it; a service is a bookable offering at a branch (e.g. a haircut) with a price and duration.
Cloudflare Workers
Code that runs on Cloudflare's global edge network instead of on a traditional server, starting on demand close to the user. Talash's backend is three such workers.
D1
Cloudflare's managed SQLite database — the system of record for all Talash data.
KV
Cloudflare's key-value store, used here for short-lived data: login state, rate-limit counts, and cached profiles.
R2
Cloudflare's object/file storage — where uploaded photos live.
Queue / Dead-letter queue (DLQ)
A list of background jobs waiting to be processed. Jobs that repeatedly fail move to the dead-letter queue so they can be inspected rather than silently lost.
Cron / Scheduled worker
Code that runs automatically on a clock (e.g. nightly), not in response to a user request.
Hono
The lightweight web framework the API is built with.
Drizzle
The type-safe toolkit used to define database tables and write queries in code.
Repository / Service (layers)
Internal code layers: a repository reads/writes the database; a service holds business rules. Keeping them separate makes the code testable and consistent.
JWT / Access token / Refresh token
A JWT is a signed token proving who you are. The short-lived access token authorizes each request; the longer-lived refresh token obtains a new access token without re-logging-in.
OAuth
The standard that lets users sign in with an existing Google account instead of a Talash-specific password. Talash supports Google only.
RBAC
Role-based access control — what a user may do depends on their role (customer, owner, manager, staff).
Expo / EAS
Expo is the framework the two mobile apps are built with; EAS is its build-and-submit service used to ship them to the app stores.
Next.js / OpenNext
Next.js is the framework for the two websites; OpenNext adapts a Next.js build to run on Cloudflare.
Turborepo
The tool that coordinates building, testing, and deploying the many packages in this single repository, only re-running what changed.
Workers AI
Cloudflare's on-edge AI service, used optionally to re-rank search results.

8 · What would make this documentation more accurate

This page is built strictly from the codebase and committed docs. The following were not derivable from the source and would sharpen the picture if you can provide them:

  1. Scale figures. Active users, peak requests-per-second, booking volume, and data size are unknown — no numbers were invented. These would let us add capacity/limits to the diagrams.
  2. Environments beyond staging/production. CI confirms staging and main targets; whether per-PR preview deployments or other environments exist is unconfirmed.
  3. Observability & alerting. A dead-letter queue (talash-dlq) exists, but whether it (and worker logs/metrics) is actively monitored, and how on-call/alerting works, is unknown.
  4. Email status. TALASH_EMAIL is provisioned but the OTP email path was retired. Confirm whether transactional email is planned to return, or the binding should be removed.
  5. Token TTLs & rate limits. Access (15 min) / refresh (30 day) TTLs and specific rate-limit thresholds are taken from in-repo guides; confirm they match current Worker config and secrets.
  6. Disaster recovery. Backup/restore strategy for D1 and R2, and any RPO/RTO targets, are not documented in the repo.
  7. External SLAs. Expected behaviour when Google OAuth, Expo Push, or Workers AI are degraded (timeouts, fallbacks) is partly visible in code but not stated as policy.
  8. Regional/data-residency requirements. Whether the Bangladeshi market imposes data-residency constraints (relevant given Cloudflare's global edge) is unknown.
